动态调度会影响程序的访存顺序

访存指令往往依赖于算数指令，它们承担着将计算结果写回到主存中的作用。如果之前的算数指令被乱序执行了，那么和它有数据依赖关系的访存指令也会受到影响。考虑一种简单的情况，如果两条访存指令A和B，A指令在前（即程序本来的顺序是A先访存），如果A所依赖的算数指令延迟较长（比如说是乘除指令），而B依赖的算数指令延迟较短（比如说是加减指令），那么在这种情况下，当B依赖的算数指令计算出结果之后，B就可以调度执行了，在这种情况下，本来A->B的访存顺序就变成了B->A，

MSHR会影响程序的访存顺序

在之前的试验中，我一直没深入的理解MSHR的大小到底会对程序的访存顺序产生什么影响。今天付博提到一个问题：L1 Dcache的miss数目是不是就是L2 cache的访问数量？
咋一听好像是这样的。因为只有L1缺失了之后才会去访问L2。这句话没错，但是这并不代表L1的缺失数量等于L2的访存数量，这两个不是一一对应的关系，为什么？因为MSHR的存在！
MSHR的作用就是当L1 cache miss时，会先检查MSHR，看是否已经有相同的数据请求（访问同一个cacheline的请求）已经发出过，如果确实发出过相同的数据请求的话，就将当前的请求合并到之前的相同的请求中。这种做法可能导致的一个结果就是多个L1 cache miss对应于一个L2 cache access。相关的实验现象如下：

可以看到：
dcache miss + dcache hit = dcache access
L2 cache miss + L2 cache hit = L2 access
但是：dcache miss = 53107 >>>> L2 cache access = 7330;
产生这种结果的原因正如我们上面所讨论的那样；

分支预测也会影响访存的顺序

分支预测对访存的影响，这一点是师兄的论文以及之前讨论的五种乱序机制（预取，动态调度，speculative load，并发访存，load in store）所没有考虑的。这一点是我在看《超标量处理器设计》（姚永斌）这本书P318时想到的，书中指出，“对于一个发生缺失的数据块来讲，如果访问这个数据块的所有load、store指令都处于分支预测失败的路径上，那么即使这个数据块被从下一级存储器中取出来，也不应该被写到Dcache中，这样可以保护Dcache不会受到这些预测错误的分支指令的影响”，从这句话中显然可以得知，分支预测错误会导致一些“无用”的访存（即程序原本不应该有这些访存事件，但是由于分支预测错误，而导致了这些访存事件的发生）
　　ps：一条指令到达流水线的提交阶段（此时这条指令在ROB中被标记为完成（complete)状态）并不代表这条指令就一定是正确的，由于分支预测失败和异常等原因，一条处于已完成状态的指令很可能还会从流水线中被抹掉，只有在这条指令之前进入到流水线中的所有指令都已经退休了，并且这条指令已经处于完成的状态的时候，它才可以退休而离开流水线，通过这样的方式使程序在超标量处理器中执行的时候有串行的效果。一条指令的生命周期是这样的：当进入到commit阶段时，会将这条指令在ROB中的状态标记为complete的状态，当这条指令前面的指令都retired了，并且其自己是completed，那么此时这条指令也可以retire了。
